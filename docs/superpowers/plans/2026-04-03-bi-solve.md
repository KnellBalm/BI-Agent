`context/playbooks/diagnostic.md`:

```markdown
# 진단 (Diagnostic) 분석 Playbook

> "왜 X가 바뀌었나?" 유형의 문제에 적용한다.

## 언제 사용하나

- 특정 지표가 갑자기 오르거나 내렸을 때
- 예상과 다른 결과가 나왔을 때
- "왜", "원인", "이유"가 질문에 포함될 때

## 표준 분석 단계

### 1단계: 지표 분해 (Metric Decomposition)
전체 지표를 구성 요소로 분해한다.
- 예: 매출 = 방문자 수 × 전환율 × 객단가
- 어느 구성 요소가 변했는지 확인
- 변화가 시작된 시점 특정 (날짜/이벤트)

### 2단계: 세그먼트 드릴다운
변화가 특정 세그먼트에 집중되어 있는지 확인한다.
- 채널별 (오가닉/유료/다이렉트)
- 기기별 (모바일/데스크탑)
- 지역별
- 신규/기존 고객별
- 제품/카테고리별

### 3단계: 시계열 패턴 확인
- 이전 같은 기간과 비교 (YoY, WoW)
- 추세인가, 일시적인가?
- 외부 이벤트(휴일, 마케팅 캠페인, 경쟁사)와 연관성

### 4단계: 원인 귀속 (Attribution)
가설 목록 작성 → 데이터로 하나씩 검증
- 각 가설에 대해: "이것이 원인이라면 A 지표가 B로 보여야 한다"
- 반증 가능한 형태로 가설 구성

### 5단계: 결론 및 권고
- 확인된 원인 명시
- 기각된 가설과 이유
- 다음 액션 (수정 가능한 원인이면 액션, 외부 원인이면 모니터링)

## 체크리스트

- [ ] 지표 분해 완료 — 어느 구성 요소가 변했는지 특정
- [ ] 변화 시작 시점 특정
- [ ] 최소 3개 세그먼트 드릴다운
- [ ] 이전 기간 대비 비교
- [ ] 가설 목록 작성 (최소 2개)
- [ ] 각 가설 데이터로 검증/기각
- [ ] 결론: 원인 명시 또는 "데이터 부족으로 미결"

## 흔한 함정

- **생존자 편향**: 데이터가 있는 세그먼트만 보고 없는 세그먼트 놓침
- **상관 ≠ 인과**: 같은 시점에 변한 것이 원인이 아닐 수 있음
- **집계 착시**: 전체 평균이 안정적이어도 세그먼트별로 상반된 움직임 가능 (Simpson's Paradox)

## 도메인별 변형

**SaaS:** 구성 요소 = MRR = 신규 MRR + 확장 MRR - 축소 MRR - 이탈 MRR
**커머스:** 구성 요소 = 매출 = 방문자 × 전환율 × 객단가
**B2B:** 구성 요소 = 파이프라인 = 리드 수 × MQL 전환율 × SQL 전환율 × 수주율
```

- 예외적으로 높거나 낮은 값

- 추가 분석이 필요한 가설

**SaaS:** 주요 탐색 대상 = 코호트별 리텐션 곡선, 기능별 사용 빈도, 구독 플랜 분포
**커머스:** 주요 탐색 대상 = 카테고리별 구매 빈도, 객단가 분포, 재구매 간격

# 비교 (Comparative) 분석 Playbook

> "A와 B 중 뭐가 나은가?" 유형의 문제에 적용한다.

## 언제 사용하나

- A/B 테스트 결과를 분석할 때
- 두 그룹(고객 세그먼트, 채널, 기간)을 비교할 때
- 의사결정을 위해 옵션 간 차이를 정량화할 때

## 표준 분석 단계

### 1단계: 비교 대상 정의
- A 그룹과 B 그룹의 정의를 명확히
- 비교 지표 (Primary metric) 1개 선정
- 가드레일 지표 (놓치면 안 되는 지표) 목록

### 2단계: 동질성 확인
- 두 그룹의 기본 특성이 유사한지 확인
- 샘플 크기, 기간, 구성이 비교 가능한지
- 교란 변수 식별

### 3단계: 기술 통계 비교
- 각 그룹의 평균, 중앙값, 분산
- 분포 차이 시각화 (박스플롯, 분포 겹치기)

### 4단계: 통계적 유의성 검정
- t-검정 / Mann-Whitney U (연속 변수)
- 카이제곱 검정 (범주형 변수)
- p-value 해석: p < 0.05 → 유의미한 차이
- 효과 크기 (Effect size): 통계적으로 유의해도 실질적으로 작을 수 있음

### 5단계: 실질적 의미 해석
- 차이가 비즈니스적으로 의미 있는 크기인가?
- 신뢰구간으로 불확실성 표현
- 권고: A 또는 B, 또는 "차이 없음"

## 체크리스트

- [ ] Primary metric 1개 선정
- [ ] 가드레일 지표 목록 작성
- [ ] 두 그룹 샘플 크기 확인
- [ ] 기술 통계 비교 완료
- [ ] 통계 검정 실행 (p-value 기록)
- [ ] 효과 크기 계산
- [ ] 가드레일 지표 이상 없음 확인
- [ ] 최종 권고 1문장

## 흔한 함정

- **p-hacking**: 여러 지표를 동시에 검정하면 우연히 유의미한 결과가 나올 확률 증가. Primary metric을 사전에 정할 것
- **샘플 크기 무시**: 통계 유의성은 샘플이 충분할 때만 의미 있음
- **실질적 의미 혼동**: p < 0.05 = 우연이 아님. 크게 중요함 ≠ 같은 뜻

## 도메인별 변형

**SaaS A/B 테스트:** Primary metric = 전환율 또는 리텐션. 최소 1~2주 관측 필요
**마케팅 채널 비교:** Primary metric = CAC 또는 LTV/CAC. 채널 믹스 차이 고려
**기능 비교:** 사용률, 완료율, 이탈률 비교

# 예측 (Predictive) 분석 Playbook

> "앞으로 어떻게 될까?" 유형의 문제에 적용한다.

## 언제 사용하나

- 다음 달/분기 지표를 예측해야 할 때
- 트렌드가 계속될지 판단해야 할 때
- 목표 달성 가능성을 추정해야 할 때

## 표준 분석 단계

### 1단계: 예측 목적 및 범위 정의
- 무엇을 예측하는가? (지표, 기간, 정확도 요구 수준)
- 예측이 어떤 결정에 쓰이는가?
- 허용 오차 범위 (±10%? ±30%?)

### 2단계: 과거 데이터 탐색
- 사용 가능한 데이터 기간
- 계절성 패턴 (요일, 월별)
- 트렌드 방향 (상승/하강/보합)
- 이상치 및 구조 변화 시점

### 3단계: 예측 방법 선택
**데이터 적을 때 (< 1년):** 이동평균, 선형 추세
**데이터 충분할 때 (> 1년, 계절성 있음):** 계절 분해, Holt-Winters
**정교한 예측이 필요할 때:** Prophet, ARIMA (데이터 사이언티스트 필요)

### 4단계: 예측 실행 및 검증
- 최근 데이터 일부를 검증 세트로 보류
- 모델을 과거 데이터로 학습
- 검증 세트에서 오차(MAPE) 측정

### 5단계: 결과 제시
- 점 예측(단일 값) + 예측 구간(범위)
- "X가 Y가 될 것이다"보다 "X는 Y~Z 범위일 가능성이 높다"
- 예측을 무너뜨릴 가정 및 리스크 명시

## 체크리스트

- [ ] 예측 지표와 기간 명시
- [ ] 사용 가능한 데이터 기간 확인
- [ ] 계절성/트렌드 패턴 확인
- [ ] 예측 방법 선택 및 이유 명시
- [ ] 예측 구간(범위) 포함
- [ ] 주요 가정 및 리스크 목록

## 흔한 함정

- **과적합**: 과거 패턴에 너무 맞춰 미래를 예측하면 실제와 괴리

## 이 스킬의 목적

"다음에 뭘 해야 하나"가 막막한 DA를 위해:

2. 한 번에 하나의 질문만 한다
3. 사전 질문지 작성 전에 분석을 시작하지 않는다
4. 가설 없이 분석을 완료했다고 선언하지 않는다

### [G1] 문제 분류

|---|---|---|
| 진단 | "왜", "원인", "이유", "떨어졌나", "올랐나" | `context/playbooks/diagnostic.md` |
| 탐색 | "어떻게 생겼나", "살펴보고", "분석해봐", "파악" | `context/playbooks/exploratory.md` |
| 비교 | "A vs B", "더 나은", "차이", "A/B 테스트" | `context/playbooks/comparative.md` |
| 예측 | "얼마나 될까", "전망", "예측", "앞으로" | `context/playbooks/predictive.md` |
| 결정 | "어떻게 해야", "선택", "결정", "우선순위" | `context/playbooks/decision.md` |
| 모니터링 | "대시보드", "매주 보고", "KPI", "지표 체계" | `context/playbooks/monitoring.md` |

**유형이 불명확할 때:** 소크라테스식으로 한 번에 하나의 질문으로 파악한다.

### [G2] 환경 체크

- **없음** → "데이터 없이도 진행할 수 있습니다. 수동으로 확인 가능한 것부터 시작하고, 필요한 데이터를 어떻게 구할 수 있는지 같이 생각해보겠습니다." → G3으로 진행
